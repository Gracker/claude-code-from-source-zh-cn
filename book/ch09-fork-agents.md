# 第九章：分叉代理（Fork Agent）与提示缓存（Prompt Cache）

## 百分之九十五的洞见

当父代理并行产生五个子代理时，每个子代理的 API 请求中绝大部分都是相同的。系统提示相同。工具定义相同。对话历史相同。触发产生的助理消息相同。唯一不同的是最后的指令：「你负责数据库迁移」、「你写测试」、「你更新文件」。

在典型的分叉场景中，若对话已经暖起来，共享前缀可能有 80,000 个 token。每个子代理的指令可能只有 200 个 token。那是 99.75% 的重叠。Anthropic 的提示缓存（Prompt Cache）对已缓存的输入 token 提供九折优惠。如果你能让那 80,000 个 token 在第二到第五个子代理中命中缓存，你就为那四个请求省下了 90% 的输入成本。对父代理来说，这就是在同一次并行派遣中花 $4 还是花 $0.50 的差别。

问题在于提示缓存是字节精确（byte-exact）的。不是「够相似」。不是「语义等价」。字节必须完全匹配，从系统提示的第一个字节到每个子代理内容开始分歧前的最后一个字节，逐字符比对。多一个空格、工具定义排序不同、某个过期的功能标志改变了系统提示的片段——缓存就会失效。整个前缀都会以全价重新处理。

分叉代理（Fork Agent）是 Claude Code 对此限制的解答。它们不只是「带上下文派生子代理」的便利功能——它们是伪装成编排功能的提示缓存利用机制。分叉系统中的每一个设计决策都回归到一个问题：如何保证并行子代理之间拥有字节相同（byte-identical）的前缀？

---

## 分叉子代理继承什么

分叉代理从父代理继承四项东西，而且是通过引用或字节精确复制来继承，而非重新计算。

**1. 系统提示。** 不是重新产生——而是穿透传递（threading）。父代理已经渲染好的系统提示字节通过 `override.systemPrompt` 传递，从 `toolUseContext.renderedSystemPrompt` 中取出。这是父代理最近一次 API 调用中实际发送的确切字符串。

**2. 工具定义。** 分叉代理定义声明了 `tools: ['*']`，但搭配 `useExactTools` 标志设为 true，子代理直接接收父代理已组装好的工具数组。不做筛选、不做重排、不做重新序列化。

**3. 对话历史。** 父代理与 API 交换的每一则消息——用户回合、助理回合、工具调用、工具结果——都通过 `forkContextMessages` 复制到子代理的上下文中。

**4. 思考配置与模型。** 分叉定义指定 `model: 'inherit'`，解析为父代理的确切模型。相同模型意味着相同的分词器、相同的上下文窗口、相同的缓存命名空间。

分叉代理的定义本身是极简的——几乎是一个空操作：

分叉代理的定义刻意极简——它从父代理继承一切。它指定所有工具（`'*'`）、继承父代理的模型、使用气泡模式（bubble mode）处理权限（让提示浮现在父代理的终端中），并提供一个永远不会被调用的空操作系统提示函数——真正的提示通过覆盖通道送达，已经渲染好且字节稳定。

## 字节相同前缀技巧

传送给 Claude 的 API 请求有特定结构：系统提示、然后是工具、然后是消息。要让提示缓存命中，从请求开头到某个前缀边界的每一个字节在各请求之间都必须相同。

分叉代理通过确保三个层级被冻结来达成这一点：

**第一层：通过穿透传递系统提示，而非重新计算。**

当父代理的系统提示在其最后一次 API 调用中被渲染时，结果被捕获在 `toolUseContext.renderedSystemPrompt` 中。这是经过所有动态插值之后的字符串——GrowthBook 功能标志、环境详情、MCP 服务器描述、技能内容、CLAUDE.md 文件。分叉子代理收到的正是这个确切字符串。

为什么不直接再次调用 `getSystemPrompt()` 呢？因为系统提示的产生不是纯函数。GrowthBook 标志在 SDK 取得远端配置后会从冷状态转为暖状态。在父代理第一个回合中返回 `false` 的标志，到分叉子代理启动时可能返回 `true`。如果系统提示包含一个由该标志控制的条件区块，重新渲染的提示就会产生哪怕是一个字符的差异。缓存爆掉。80,000 个 token 以全价重新处理，乘以五个子代理。

穿透传递已渲染的字节消除了这整类分歧。

**第二层：工具定义通过精确传递。**

一般的子代理（Sub-Agent）会经过 `resolveAgentTools()`，它根据代理定义的 `tools` 和 `disallowedTools` 数组筛选工具池、应用权限模式差异，并可能重新排序工具。产生的序列化工具数组会与父代理的不同——不同的子集、不同的顺序、不同的权限标注。

分叉代理完全跳过这些：

```typescript
const resolvedTools = useExactTools
  ? availableTools  // 父代理的精确数组
  : resolveAgentTools(agentDefinition, availableTools, isAsync).resolvedTools
```

`useExactTools` 标志只在分叉路径上被设为 true。子代理原封不动地取得父代理的工具池。相同的工具、相同的顺序、相同的序列化。这包括在子代理的工具池中保留 Agent 工具本身，即使子代理被禁止使用它——移除它会改变工具数组并破坏缓存。

**第三层：消息数组构建。**

这是 `buildForkedMessages()` 精心处理的部分。此函数构建位于共享历史和每个子代理指令之间的最后两则消息：

`buildForkedMessages()` 函数构建位于共享历史和每个子代理指令之间的最后两则消息。算法：

1. 复制父代理的助理消息（保留所有 `tool_use` 区块及其原始 ID）。
2. 对每个 `tool_use` 区块，建立一个带有固定占位字符串的 `tool_result`（所有子代理完全相同）。
3. 构建一则用户消息，包含所有占位结果，后面接着用样板标签包装的每个子代理指令。
4. 返回 `[clonedAssistantMessage, userMessageWithPlaceholdersAndDirective]`。

```typescript
// 伪代码——示意消息建构过程
function buildChildMessages(directive, parentAssistant) {
  const cloned = cloneMessage(parentAssistant)
  const placeholders = parentAssistant.toolUseBlocks.map(b =>
    toolResult(b.id, CONSTANT_PLACEHOLDER)  // 所有子代理字节相同
  )
  const userMsg = createUserMessage([...placeholders, wrapDirective(directive)])
  return [cloned, userMsg]
}
```

每个子代理产生的消息数组看起来像：

```
[...共享历史, assistant(所有tool_use), user(占位结果..., 指令)]
```

指令之前的每个元素在所有子代理之间都是相同的。`FORK_PLACEHOLDER_RESULT`——一个固定字符串 `'Fork started -- processing in background'`——确保即使是工具结果区块也是字节相同的。`tool_use_id` 的值是相同的，因为它们引用同一则助理消息。只有最后的文字区块——包含每个子代理的指令——是不同的。

缓存边界恰好落在那个最后的文字区块之前。它上方的一切——可能数万个 token 的系统提示、工具定义、对话历史和占位结果——在第一个子代理之后的每个子代理都以九折优惠命中缓存。

## 分叉样板标签

每个子代理的指令被包装在一个样板 XML 标签中，它有两个用途：指示子代理应如何行为，以及作为递归分叉侦测的标记。

样板包含大约 10 条规则。关键的几条：

- **覆盖父代理的分叉指令。** 父代理的系统提示说「默认使用分叉」——样板明确告诉子代理：「那条指令是给父代理的。你就是分叉。不要派生子代理。」
- **静默执行，报告一次。** 工具调用之间不产生对话文字。直接使用工具，然后产出结构化摘要。
- **保持在范围内。** 子代理不得超出其指令范围。
- **结构化输出格式。** 响应必须遵循 Scope/Result/Key files/Files changed/Issues 的范本，当多个子代理同时回报时，让父代理容易解析结果。

规则 1 特别有趣。父代理的系统提示——分叉子代理为了缓存原因而逐字继承的——包含类似「当有并行工作时默认使用分叉」的指令。如果子代理遵循那条指令，它会尝试分叉自己的子代理，造成代理的无限递归。样板明确覆盖：「那条指令是给父代理的。你就是分叉。」

结构化输出格式（Scope/Result/Key files/Files changed/Issues）不是装饰性的。它将子代理的输出限制在事实性报告中，这让父代理在五个子代理同时回报时更容易解析和汇整结果。

## 递归分叉防护

分叉子代理在其工具池中保留了 Agent 工具。它必须这样做——移除它会改变序列化的工具数组并破坏提示缓存。但如果子代理实际上在没有指定 `subagent_type` 的情况下调用了 Agent 工具，分叉路径会再次触发，产生一个孙代理分叉。这个孙代理会继承更大的上下文（父代理 + 子代理的对话），产生自己的分叉，如此反复。

两道防护机制阻止了这个情况：

**主要防护：querySource 检查。** 当分叉子代理被产生时，其 `context.options.querySource` 被设为 `'agent:builtin:fork'`。`call()` 方法在允许分叉路径之前会检查这个值：

```typescript
// 在 AgentTool.call() 中：
if (effectiveType === undefined) {
  // 分叉路径——但我们已经在分叉中了吗？
  if (querySource === 'agent:builtin:fork') {
    // 拒绝：已经是分叉子代理
  }
}
```

这是快速路径。它只检查选项对象中的一个字符串。

**备用防护：消息扫描。** 分叉防护使用两道防线：在产生时设置的 `querySource` 标记（快速路径——单次字符串比对），以及扫描消息历史中是否有样板 XML 标签的备用机制。备用机制存在是因为 `querySource` 能存活过自动压缩（autocompact），但在某些它未被正确传递的边界情况下，消息扫描备用能捕捉到递归。这是一种双重保险的做法，其中检查的成本（扫描消息）相较于意外递归分叉的成本（失控的 API 开销）微乎其微。

为什么需要备用？因为 Claude Code 有一个自动压缩（autocompact）功能，当上下文过长时会重写消息数组。自动压缩能重写消息内容但会保留选项中的 `querySource`。理论上，`querySource` 本身就足够了。实务上，消息扫描备用捕捉了 `querySource` 未被正确传递的边界情况——这是一种双重保险的做法，其中检查的成本（扫描消息）相较于意外递归分叉的成本（失控的 API 开销）微乎其微。

## 同步到异步转换

分叉子代理一开始在前景执行：它的消息流到父代理的终端，父代理阻塞等待完成。但如果子代理花太长时间呢？Claude Code 允许执行中转为后台——用户（或自动超时机制）可以将正在执行的前景代理推到后台，而不会遗失任何工作。

机制出乎意料地干净：

1. 当前景代理通过 `registerAgentForeground()` 注册时，会建立一个后台信号 promise。

2. 父代理的同步循环在代理的消息流和后台信号之间竞争：

```
while (true) {
  const result = await Promise.race([
    iterator.next(),         // 代理的下一则消息
    backgroundSignal,        // 「移到后台」触发器
  ])
  if (result === BACKGROUND_SIGNAL) break
  // ... 处理消息
}
```

3. 当后台信号触发时，前景迭代器通过 `iterator.return()` 优雅地终止。这会触发生成器的 `finally` 区块来处理清理工作。

4. 一个新的 `runAgent()` 实例以 `isAsync: true` 产生，使用相同的代理 ID 和截至目前累积的消息历史。代理从中断处继续，现在在后台执行。

5. 原本的同步 `call()` 返回 `{ status: 'async_launched' }`，父代理继续其对话。

不会遗失任何工作，因为消息历史就是代理的状态。磁盘上的侧链转录本（sidechain transcript）记录了代理产出的每一则消息。新的异步实例从这份转录本回放，从同步实例停止的地方继续。

## 自动后台化

当 `CLAUDE_AUTO_BACKGROUND_TASKS` 环境变量或 `tengu_auto_background_agents` GrowthBook 标志启用时，前景代理会在 120 秒后自动被后台化：

当通过环境变量或功能标志启用时，前景代理会在 120 秒后自动被后台化。禁用时，函数返回 0（不自动后台化）。

这是一个带有成本影响的用户体验决策。前景代理会阻塞父代理的终端——用户无法输入、无法发出新指令、无法产生其他代理。两分钟足够让代理同步完成大部分快速任务（此时流输出是有用的反馈），但又短到长时间执行的任务不会绑架终端。

在分叉实验下，自动后台化问题是多余的：所有分叉产生的代理从一开始就强制异步。`run_in_background` 参数完全从 schema 中隐藏。每个分叉子代理都在后台执行，完成后通过 `<task-notification>` 回报，父代理永远不会阻塞。

## 何时不使用分叉

分叉是数种编排模式之一，在三种情况下会被刻意排除：

**协调者（Coordinator）模式。** 协调者模式和分叉模式互斥。协调者有结构化的委派模型：它维护计划、以明确的提示指派任务给工作者、并追踪进度。分叉的「继承一切」做法会破坏这个模型。被分叉的协调者会继承父协调者的系统提示（上面写着「你是协调者，委派工作」），子代理会尝试编排而非执行。`isForkSubagentEnabled()` 函数会先检查 `isCoordinatorMode()`，如果启用则返回 false。

**非交互式会话。** SDK 和 API 消费者（`--print` 模式、Claude Agent SDK）在没有终端的情况下运作。分叉的 `permissionMode: 'bubble'` 会将权限提示浮现到父代理的终端——在非交互式模式中不存在。与其构建一个独立的权限流程，分叉路径直接被禁用。SDK 消费者改用明确的 `subagent_type` 选择。

**明确指定 subagent_type。** 当模型指定了 `subagent_type`（例如 `"Explore"`、`"Plan"`、`"general-purpose"`），分叉路径不会被触发。分叉只在 `subagent_type` 被省略时触发。这让模型可以在「我想要一个有自己系统提示和工具集的专用代理」（明确指定类型）和「我想要一个继承我上下文的自身复制品来并行处理这件事」（省略类型）之间做选择。

## 经济学

考虑一个具体场景。开发者要求 Claude Code 重构一个模块。父代理分析了代码库、形成计划，并并行派遣五个分叉子代理：一个更新数据库 schema、一个重写服务层、一个更新路由器、一个修复测试、一个更新类型。

在对话的这个时间点，共享上下文相当大：
- 系统提示：约 4,000 个 token
- 工具定义（40+ 个工具）：约 12,000 个 token
- 对话历史（分析 + 计划）：约 30,000 个 token
- 带有五个 tool_use 区块的助理消息：约 2,000 个 token
- 占位工具结果：约 500 个 token

共享前缀总计：约 48,500 个 token。每个子代理指令：约 200 个 token。

不使用分叉（五个独立代理，各自有全新上下文和自己的系统提示）：
- 每个子代理处理自己的系统提示 + 工具 + 任务提示
- 没有缓存共享（不同的系统提示、不同的工具集）
- 成本：5 x 全价输入处理

使用分叉（字节相同的前缀）：
- 子代理 1：48,700 个 token 以全价处理（第一个请求缓存未命中）
- 子代理 2-5：48,500 个 token 以 10% 价格处理（缓存命中）+ 每个 200 个 token 以全价处理
- 子代理 2-5 的有效成本：约 4,850 + 200 = 每个约 5,050 个 token 等值

节省的金额随上下文大小和子代理数量而增长。对于一个拥有 100K token 历史的暖会话派生 8 个并行分叉，缓存节省可以超过未共享时输入 token 成本的 90%。

这就是为什么分叉系统中的每一个设计决策——穿透传递而非重新计算、精确工具传递、占位结果、甚至在子代理的工具池中保留被禁止使用的 Agent 工具——都在为一件事优化：字节相同的前缀。每个决策都用少量的优雅或安全性来换取可衡量的 API 成本降低。

## 设计张力

分叉系统做出了值得理解的明确取舍：

**隔离性 vs. 缓存效率。** 分叉子代理继承一切，包括可能与其任务无关的对话历史。一个重写测试的子代理不需要父代理讨论数据库 schema 设计的那 15 则消息。但包含那些消息正是让前缀相同的关键。剔除无关历史可以节省上下文窗口空间，但代价是破坏缓存。设计上的赌注是缓存节省超过上下文开销。

**安全性 vs. 缓存效率。** Agent 工具留在分叉子代理的工具池中，即使子代理不得使用它。移除它会更安全（子代理甚至无法尝试分叉），但会改变工具数组的序列化。样板标签和递归分叉防护是补偿控制——用运行时防护代替静态移除。

**简洁性 vs. 缓存效率。** 占位工具结果是一个谎言。子代理对于父代理助理消息中的每个 `tool_use` 区块都看到 `'Fork started -- processing in background'`，不管那些工具调用实际上做了什么。这没问题，因为子代理的指令告诉它要做什么——它不需要父代理派遣回合中准确的工具结果。但这意味着子代理的对话历史在技术上是不连贯的。占位文字的选择是为了简短和一致性，而非准确性。

这些取舍中的每一个都反映了同样的优先级：当你在规模化使用 API 调用时按 token 付费，字节相同的前缀值得你为此扭曲架构。

## 实践应用：为提示缓存效率而设计

分叉代理模式的适用范围超越 Claude Code。任何从相同上下文派遣多个并行 LLM 调用的系统都能从缓存感知的请求构建中获益。原则如下：

**1. 穿透传递已渲染的提示，不要重新计算。** 如果你的系统提示包含任何动态内容——功能标志、时间戳、用户偏好、A/B 测试变体——捕获渲染结果并以值传递给子代理。重新计算有分歧的风险。

**2. 冻结工具数组。** 如果你的子代理需要不同的工具集，你就放弃了工具区块上的缓存共享。考虑保留完整的工具集，使用运行时防护（像分叉样板中的「不要使用 Agent」）来代替编译时期的移除。

**3. 最大化共享前缀，最小化每个子代理的后缀。** 结构化你的消息数组，让所有共享内容排在前面，每个子代理的内容附加在末尾。交错混合共享和每个子代理的内容会碎片化缓存边界。

**4. 对可变内容使用固定占位符。** 当消息结构需要响应先前的工具调用时，在所有子代理中使用相同的占位字符串，而非实际的（会分歧的）结果。

**5. 衡量损益平衡点。** 缓存共享有其开销：每个子代理更大的上下文窗口（它们携带无关历史）、运行时防护代替静态安全性、架构复杂度。计算你的并行模式（多少个子代理、共享前缀多大）在扣除额外上下文 token 后是否确实省钱。
