# 第十八章：我们学到了什么

## 五个架构赌注

Claude Code 不是唯一的代理系统，也不是最早的。但它做出了五个架构赌注，使其在代理框架的版图中脱颖而出，经过近两千个文件和十七个章节之后，这些赌注值得仔细检视。

### 赌注一：生成器循环胜过回调

大多数代理框架给你一个管道：定义工具、注册处理器、让框架进行编排。开发者编写回调。框架决定何时调用它们。

Claude Code 做的恰恰相反。`query()` 函数是一个异步生成器——开发者拥有循环的控制权。模型流响应，生成器 yield 工具调用，调用者执行它们、附加结果，然后生成器循环。只有一个函数、一个数据流、一个所有互动都会通过的地方。生成器返回类型的 10 个终止状态和 7 个延续状态编码了每一种可能的结果。循环就是系统本身。

这个赌注是：即使一个生成器函数成长到 1,700 行，它仍然比分散的回调图更容易理解。研究过源码之后，这个赌注得到了回报。当你想了解一个会话为什么结束时，你只需看一个函数。当你想新增一个终止状态时，你只需在一个可辨识联集（discriminated union）中新增一个变体。类型系统强制执行穷举处理。回调架构会将这些逻辑分散在数十个文件中，而回调之间的交互会是隐式的，而不是在控制流程中可见的。

### 赌注二：基于文件的记忆胜过数据库

第十一章已经详细阐述了这个论点，但架构的重要性远超记忆系统本身。选择使用纯 Markdown 文件而非 SQLite、向量数据库或云端服务，是一个押注透明度胜过功能的赌注。数据库可以支持更丰富的查询、更快的查找和交易保证。文件什么都没有。文件提供的是信任。

一个用户打开 `~/.claude/projects/myapp/memory/MEMORY.md` 用 vim 检视，看到代理记住了关于他们的确切内容——这和必须问代理「你记得什么？」然后期望答案是完整的用户，与系统之间有着根本不同的关系。基于文件的设计让代理的知识状态可以被外部观察到，而不仅仅是自我报告的。这比查询性能更重要。LLM 驱动的回忆系统以检索智慧补偿了存储的简单性——一个 Sonnet 侧查询从清单中精选五条相关记忆，比嵌入相似度更精确，而且不需要任何基础设施。

### 赌注三：自我描述的工具胜过中央编排器

代理框架通常提供一个工具注册表：你在一个中央配置中描述你的工具，框架将它们呈现给模型。Claude Code 的工具自我描述。每个 `Tool` 对象携带自己的名称、描述、输入 schema、提示词贡献、并行安全标志和执行逻辑。工具系统的职责不是向模型描述工具——而是让工具描述自己。

这个赌注在可扩展性上获得了回报。MCP 工具（第十五章）通过实现相同的接口成为一等公民。来自 MCP 服务器的工具和内置工具对模型而言无法区分。系统不需要一个独立的「MCP 工具转接器」层——包装产生一个标准的 `Tool` 对象，从那一刻起，现有的工具管道就能处理它：权限检查、并行执行、结果预算控制、钩子拦截。

### 赌注四：分叉代理实现缓存共享

第九章涵盖了分叉机制：一个子代理从父代理的完整对话开始，在其上下文窗口中共享父代理的提示缓存。这不是一个方便性的优化——这是一个架构赌注，赌缓存共享模型值得承担分叉生命周期管理的复杂性。

替代方案——用对话摘要产生一个全新的代理——更简单但更昂贵。每个全新的代理都必须从头付出处理其上下文的全部成本。一个分叉代理可以免费获得父代理的缓存前缀（输入 token 享有 90% 折扣），这使得为小型任务产生代理变得经济实惠：记忆提取、代码审查、验证流程。后台记忆提取代理（第十一章）在每次查询循环转换后执行，而它的成本之所以微乎其微，正是因为它共享了父代理的缓存。没有基于分叉的缓存共享，那个代理的成本将高得令人却步。

### 赌注五：钩子胜过插件

大多数可扩展性系统使用插件——注册功能并在宿主进程中执行的代码。Claude Code 使用钩子——在生命周期节点执行的外部进程，通过退出码和 stdin/stdout 上的 JSON 进行通信。

这个赌注是：进程隔离值得付出额外开销。插件可以让宿主崩溃。钩子只会让自己的进程崩溃。插件可以将内存泄漏到宿主的堆积中。钩子的内存会随其进程一起消亡。插件需要一个必须进行版本控管和维护的 API 表面。钩子只需要 stdin、stdout 和一个退出码——这个协议从 1971 年以来一直很稳定。

额外开销是实际存在的：每次钩子调用产生一个进程需要数毫秒，而进程内回调则不需要。第十二章中内部回调的 -70% 快速路径表明系统知道这个成本很重要。但对于外部钩子——用户脚本、团队 linter、企业策略服务器——隔离保证使系统的扩展更安全。企业可以部署基于钩子的策略执行，而不用担心一个格式错误的钩子脚本会让开发者的会话崩溃。

## 什么可以迁移，什么不行

并非 Claude Code 中的每个模式都能普遍推广。有些是规模、资源或特定约束条件的结果，其他代理开发者未必会面临相同的情况。

### 可迁移到任何代理的模式

**生成器循环模式。** 任何需要流响应、处理工具调用和管理多种终止状态的代理，都能从将循环显式化而非隐藏在回调之后获益。可辨识联集返回类型——精确编码循环停止的原因——是一个能消除整类「代理为什么停了？」调试场景的模式。

**带有 LLM 回忆的基于文件记忆。** 具体的实现细节是 Claude Code 特有的，但原则——简单存储结合智慧检索——适用于任何需要跨会话持久化知识的代理。四类型分类法（用户、反馈、项目、参考）和可衍生性测试（「这能否从当前项目状态重新衍生出来？」）是可重复使用的设计启发。

**远端执行的非对称读写通道。** 当读取是高频流而写入是低频 RPC 时，将它们分开是正确的，无论具体的传输协议为何。

**搜索的位图预先过滤器。** 任何在大型文件索引中搜索的代理都能从 26 位字母位图作为预先过滤器中获益。每个条目四字节，每个候选项一次整数比较——成本效益比非常出色。

**提示缓存稳定性作为架构考量。** 如果你的代理使用带有提示缓存的 API，将稳定内容放在前面、易变内容放在后面来组织提示词，这不是优化——而是一个决定你成本结构的架构决策。

### Claude Code 规模特有的模式

**分叉的终端渲染器。** Claude Code 分叉了 Ink 并使用打包类型数组、基于池的驻留和单元格层级差异比对重新实现了渲染管道，因为它需要在终端中达到 60fps 的流速度。大多数代理渲染到 Web 接口或简单的日志输出。这种工程投资只有在终端渲染是你的主要 UI 且你在高频率流时才有意义。

**50 多个启动分析检查点。** 当你有数十万用户且 0.5% 的取样能产生具统计显着性的数据时才有意义。对于较小的代理，更简单的计时系统就足够了。

**八种 MCP 传输类型。** Claude Code 支持 stdio、SSE、HTTP、WebSocket、SDK、两种 IDE 变体和一个 Claude.ai 代理，因为它必须与每种部署拓扑整合。大多数代理只需要 stdio 和 HTTP。

**钩子快照安全模型。** 在启动时冻结钩子配置且永远不隐式重新读取，是对特定威胁的防御：恶意仓库代码在用户接受信任对话框后修改钩子。这在你的代理在具有不受信任的 `.claude/` 配置的任意仓库中执行时很重要。只在受信任环境中执行的代理可以使用更简单的钩子管理。

## 复杂性的代价

近两千个文件。这带来了什么，又付出了什么代价？

文件数量作为复杂性指标具有误导性。其中很大一部分是测试基础设施、类型定义、配置 schema 和分叉的 Ink 渲染器。实际的行为复杂性集中在少数高密度文件中：`query.ts`（1,700 行，代理循环）、`hooks.ts`（4,900 行，生命周期拦截系统）、`REPL.tsx`（5,000 行，交互式编排器）以及记忆系统的提示词组建函数。

复杂性来自三个来源，各有不同的特质：

**协议多样性。** 支持五种终端键盘协议、八种 MCP 传输类型、四种远端执行拓扑和七种配置范围本身就是复杂的。每个额外的协议是对程序库的线性增加，而非指数级的——但总和很大。这种复杂性在 Brooks 的意义上是偶然的：它来自环境（终端碎片化、MCP 传输演进、远端部署拓扑），而不是来自正在解决的问题。

**性能优化。** 基于池的渲染、位图搜索预先过滤器、黏性缓存闩锁（Sticky Cache Latch）和推测性工具执行各自增加了复杂性，以换取可衡量的性能提升。这种复杂性因度量而合理化——每项优化之前都有识别出瓶颈的性能分析数据。风险在于优化会累积并以使热路径更难修改的方式相互作用。

**行为调校。** 记忆系统的提示指令、过时警告、验证协议、「忽略记忆」反模式指令——这些不是代码复杂性。它们是提示词复杂性，承载着不同的维护负担。当模型的行为在版本之间发生变化时，通过评估精心调校的提示指令可能需要重新调校。评估基础设施（在整个程序库中以案例编号和评估分数引用）是防止退化的防线，但需要持续投入。

这个系统的维护负担很重。一个阅读程序库的新工程师不仅必须理解代码路径，还必须理解促成特定提示措辞的评估结果、促成特定安全检查的生产事故，以及促成特定优化的性能概况。代码注解很详尽——许多包含评估案例编号和前后测量——但近两千个文件中详尽的注解本身也是一个阅读负担。

## 代理系统的未来走向

从 Claude Code 的模式中可以看到四个趋势，它们指向了这个领域的发展方向。

### MCP 作为通用协议

第十五章描述了 Claude Code 作为最完整的 MCP 客户端之一。重要的不是 Claude Code 的实现——而是 MCP 的存在本身。一个标准化的工具发现和调用协议意味着为一个代理构建的工具可以与任何代理一起工作。生态系统效应是显而易见的：一个 Postgres 的 MCP 服务器一旦建好，就能服务每个使用 MCP 的代理。开发者在工具整合上的投资是可携带的。

对代理开发者的启示：如果你正在定义自定义工具协议，你可能犯了一个错误。MCP 足够好了，它在不断改进，而标准协议的生态系统优势会随时间复利增长。构建一个 MCP 客户端，为规范做出贡献，让协议通过社群反馈来演进。

### 多代理协调

Claude Code 的子代理系统（第八章）、任务协调（第十章）和分叉机制（第九章）是多代理模式的早期实现。它们解决了特定问题——缓存共享、并行探索、结构化验证——但它们也揭示了根本性的挑战：协调开销。

代理之间的每条消息都消耗 token。每个分叉共享一个缓存但增加了一个对话分支，父代理最终必须调和。任务系统的状态机（排队中、执行中、已完成、已失败、已取消）是增加复杂性而不增加能力的协调机制。随着代理变得更有能力，压力将从「我们如何协调多个代理？」转向「我们如何让一个代理足够有能力以至于不需要协调？」

目前的证据表明两种方法将共存。简单的任务将使用单一代理。复杂的任务将使用协调式多代理系统。工程挑战是让协调开销足够低，使得交叉点对真正的并行工作有利于多代理，而不仅仅是对复杂的任务有利。

### 持久化记忆

Claude Code 的记忆系统是持久化代理记忆的第一版。基于文件的设计、四类型分类法、LLM 驱动的回忆、过时系统以及用于长时间执行会话的 KAIROS 模式，都是一个将显着演进的问题的第一代解决方案。

未来的记忆系统很可能会增加结构化检索（当前系统检索整个文件；未来系统可能检索特定事实）、跨项目迁移学习（到处适用的用户偏好、不适用的项目惯例）和协作记忆（第十一章的团队记忆是第一步，但同步、冲突解决和访问控制都是最小化的）。

开放性问题是基于文件的方法是否能扩展。在每个项目 200 条记忆时，它可以运作。在每个项目 2,000 条记忆时，Sonnet 侧查询的清单变得太大，整合变得太昂贵，索引超过其上限。基于文件而非数据库的架构赌注将在使用量增长时面临最严峻的考验。

### 自主运作

KAIROS 模式、后台记忆提取代理、自动梦境整合、推测性工具执行——这些都是走向自主运作的步伐。代理在不被要求的情况下做有用的工作：它记住了你忘了告诉它去记的东西，它在你睡觉时整合自己的知识，它在当前响应完成之前就开始执行下一个工具。

轨迹是清晰的。未来的代理将更少被动、更多主动。它们会注意到用户未描述的模式，建议用户未要求的修正，并在没有明确 `/remember` 指令的情况下维护自己的知识。Claude Code 的记忆系统，以其后台提取安全网和精心设计的「该存储什么」启发式提示词，是这个未来的原型。

约束是信任。自主运作需要用户信任代理在无人值守时会做正确的事。基于文件的记忆、可观察的钩子系统、过时警告、权限对话框——所有这些存在的原因是信任必须被赢得，而不是被假定。通往更自主代理的道路要经过更透明的代理。

## 结语

十七个章节。六个核心抽象。一个生成器循环在中心，工具向外延伸，记忆向后穿越时间，钩子守卫边界，一个渲染引擎将一切转译为屏幕上的字符，而 MCP 将它连接到程序库之外的世界。

Claude Code 中最深层的模式不是任何单一技术。而是一个反复出现的决策：将复杂性推向边界。渲染系统将复杂性推向池和差异比对——在管道内部，一切都是整数比较。输入系统将复杂性推向分词器和键绑定解析器——在处理器内部，一切都是类型化的动作。记忆系统将复杂性推向写入协议和回忆选择器——在对话内部，一切都是上下文。代理循环将复杂性推向终止状态和工具系统——在循环内部，只是：流、收集、执行、附加、重复。

每个边界吸收混沌并输出秩序。原始字节变成 `ParsedKey`。Markdown 文件变成被回忆的记忆。MCP JSON-RPC 变成 `Tool` 对象。钩子退出码变成权限决策。在每个边界的一侧，世界是混乱的——五种键盘协议、脆弱的 OAuth 服务器、过时的记忆、不受信任的仓库钩子。在另一侧，世界是有类型的、有边界的、且被穷举处理的。

如果你在构建代理系统，这就是可迁移的教训。不是那些具体的技术——你可能不需要基于池的渲染、KAIROS 模式或八种 MCP 传输方式。但是原则是：定义你的边界，在那里吸收复杂性，并保持它们之间的一切干净。边界是工程困难的地方。内部是工程令人愉快的地方。为愉快的内部而设计，并将你的复杂性预算投资在边缘。

源码是开放的。螃蟹在牠的蟹螯里握着地图。去读吧。
