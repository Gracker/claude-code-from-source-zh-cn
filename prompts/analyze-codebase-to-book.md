# 提示词：将任何代码库转化为技术书籍

一套可重复使用的系统，用于分析代码库并通过并行 AI 代理产出出版质量的技术书籍。设计目标是产出可与 O'Reilly 技术书媲美的成品。

---

## 提示词内容

```
我希望你分析位于 [path] 的源码，并产出一本关于其架构、模式和内部机制的
全面技术书籍。

这本书的阅读体验应如同专业技术出版物——那种资深工程师会购买来深入理解一个
系统的书。不是文件。不是教学。而是一本指导系统如何运行、每个决策为何如此做出、
以及读者可以借鉴哪些模式用于自己项目的书。

---

## 第一阶段：探索

启动平行代理（每个主要子系统一个），exhaustive 地阅读代码库中的每个文件。
每个代理应记录：

- 架构与模块边界
- 关键抽象（类型、接口、核心类）
- 数据流（信息如何在系统中流动）
- 设计模式（使用了哪些模式以及为什么）
- 整合点（此模块如何与其他模块连接）
- 出人意料的决策（任何非显而易见或巧妙的设计）

为每个子系统产出一份原始分析文件。这些是研究笔记，不是最终书籍。

## 第二阶段：受众与定位

在组织书籍结构之前，先定义：

**主要受众**：这本书是写给谁的？他们已经知道什么？他们想学什么？
本书应同时服务两类读者：
- 技术主管，想了解架构与设计背后的理由（可以跳过代码区块和深入探讨段落）
- 资深工程师，想获得实现层级的理解（阅读所有内容，包括深入探讨）

**核心论点**：关于这个系统，那个唯一的重大洞见是什么？每一章都应回扣这个
论点。对于一个代码库，这通常是：「这是这个系统所下的架构赌注，而这是每个
子系统如何服务于这个赌注。」

**为什么值得写成一本书**：为什么读者不能直接读源码就好？本书的价值在于：
叙事（源码没有叙事）、横切面模式（散落在源码的各个文件中）、设计理由
（完全不在代码里）、以及可转移的经验（需要源码无法提供的综合分析）。

## 第三阶段：结构

像读者要从零开始建构这个系统一样来组织书籍。每一章应解决一个明确的问题，
而下一章依赖于此问题的解答。读者不应遇到需要后面章节才能理解的概念。

**部分**：将章节分组为 5-7 个主题部分。每个部分有一行题词来界定该段落。
部分创造自然的阅读中断点，并让目录更易于浏览。

**章节排序原则**：
- 基础先行（启动、状态、与外部服务的通讯）
- 核心循环次之（主要执行循环——这通常是最重要的一章）
- 建构于核心之上的能力（工具、外挂、扩展）
- 进阶模式（多代理、编排、协调）
- 支持基础设施（UI、网络、持久化）
- 效能与优化放最后（你无法优化你不理解的东西）
- 结语：综合、可转移的经验、展望未来

**章节篇幅**：目标每章 300-800 行。若一章超过 800 行，代表它试图涵盖太多
——拆分它。若一章不到 200 行，可能不足以成为独立章节——合并它。

呈现完整大纲，包含部分名称、章节标题，以及每章 2-3 个描述其涵盖内容的要点。
在开始撰写前取得核准。

## 第四阶段：撰写

以第一阶段的分析作为研究笔记，从头开始撰写每一章。不要重新组织分析——
将其改写为叙事散文。

### 章节范本

每一章遵循此结构：

1. **开场**（2-3 段）
   - 这个层次/子系统解决什么问题？
   - 它为何存在？没有它会出什么问题？
   - 它如何与读者已知的内容连结？（明确回溯引用前一章）
   - 读完后读者将理解什么？

2. **正文**（核心内容）
   - 混合散文、图表、代码片段和表格
   - 散文用于叙事和理由（「为什么」）
   - 图表用于架构、数据流和状态机
   - 代码用于关键模式（仅限虚拟码——见下方规则）
   - 表格用于参考数据（栏位列表、配置选项）

3. **深入探讨段落**（选用，行内呈现）
   - 技术主管可以跳过的实现细节补充段落
   - 包含「这在位元层级上到底如何运行」的内容
   - 应可独立阅读而不影响章节的叙事主线

4. **「实践应用」**（结尾段落）
   - 恰好 5 个从本章萃取的可转移模式
   - 每个模式：名称 → 解决什么问题 → 如何调适 → 需注意的陷阱
   - 具体到足以付诸行动，抽象到足以转移至其他系统
   - 各章之间稍微变化格式以避免单调

### 语气与风格

- **专家同侪**：像一位资深工程师为同事进行深度技术审查。不是学术的，
  不是教学的，不是行销的。
- **直接且有主见**：「这很巧妙，因为……」/「这是错误的抽象，因为……」/
  「这之所以存在是因为……」
- **不灌水**：每个句子都在教某些东西，或为下一个要教的东西做铺垫。
  如果一个句子无法证明其存在价值，就删掉它。
- **展示取舍**：不要只描述建构了什么。解释什么没有被建构以及为什么。
  未走之路往往更具启发性。

### 代码区块

- **仅限虚拟码**：永远不要原封不动地复制源码。展示的是模式，不是实现。
- **每章最多 3-5 个区块**：每个区块应为 5-15 行。
- **使用不同的变量名称**：使用能说明概念的通用名称，而非源码中的确切识别字。
- **标注为示意用途**：加入如 `// 虚拟码——说明此模式` 或
  `// 为清晰起见做了简化` 的注解。
- **前置上下文**：在区块前用一句话说明它展示什么。在区块后用一段话说明为什么
  这个模式重要。

### 图表

- **Mermaid 格式**：使用 ```mermaid 围栏代码区块。这些在 GitHub 和大多数
  网页框架中都能原生渲染。
- **每个架构概念都要有图表**：数据流、状态机、决策树、时序图、组件关系图。
- **使用的图表类型**：
  - `graph TD` / `graph LR` 用于架构和数据流
  - `sequenceDiagram` 用于请求/响应流程和生命周期
  - `stateDiagram-v2` 用于状态机
  - `flowchart TD` 用于决策树和管道
  - `gantt` 用于时序图和并行执行
- **每章 2-4 张图表**：复杂章节（核心循环、工具）多一些，聚焦型章节少一些。

### 交叉引用

- 每一章的开头都要明确回指前一章
- 当某概念将在后面展开时使用前向引用：「第 N 章将深入探讨此主题」
- 每个概念只有一个正式归属——其他章节引用，而非重新解释

### 一致性检查

- 各章之间不重复修辞用语
- 标准化的「实践应用」格式（每章 5 个模式）
- 不写确切的文件数量或版本号（会过时）
- 全书术语一致

## 第五阶段：编辑审查

启动 2-3 个审查代理，各覆盖书籍的一个段落。每个审查者评估：

1. **开场质量**：是否吸引人？是否与前一章连结？
2. **流畅度**：是否有拖沓、重复、或仅列举事实而未建构洞见的段落？
3. **内容删减**：哪些是不服务于叙事的参考手册式内容？哪些代码区块太长？
4. **缺漏内容**：读者会感到困惑的缺口？缺少的过渡段落？
5. **需要的图表**：指出哪些地方可以用图表取代大段文字。详细描述每张图表。
6. **跨章一致性**：语气、格式、术语、矛盾之处。
7. **具体修改**：5-10 个需要重写的句子/段落，附上理由。

将所有审查反馈汇整为一份排序过的行动计划。

## 第六阶段：修订

一次性套用所有审查反馈：

- **结构调整**：拆分/合并章节、修复断裂的引用、必要时补上遗漏的结尾章节
- **去重**：每个概念只解释一次，其他地方使用交叉引用
- **内容删减**：移除冗长列举（保留模式）、精简膨胀的段落、
  将参考数据性质的内容压缩成表格
- **内容增补**：实现范例、真实世界的钩子范例、在标记位置加入图表
- **一致性**：标准化「实践应用」段落、修复重复用语、验证交叉引用

## 第七阶段：源码审计

出版前，将每个代码区块与原始源码进行比对审计：

- **替换**任何逐字或近逐字复制的区块，改用不同变量名称的虚拟码
- **标注**类型签名，加上「// 示意用途」注解
- **验证**没有残留专有提示词文字、内部常数或确切的函数实现

本书教的是模式和架构。它不应让人得以重建确切的源码。
```

## 如何使用的

此提示词产出了 **《Claude Code from Source》**——18 章分析 Anthropic AI 编码代理架构的书籍，通过 npm 泄漏的源码映射（Source Map）进行逆向工程。

**数据概览：**
- 36 个 AI 代理，横跨 7 个阶段
- 总制作时间约 6 小时
- 第一阶段：6 个探索代理阅读了 1,884 个源码文件
- 第二至三阶段：组织为 7 个部分、18 个章节
- 第四阶段：15 个编写代理产出 10,320 行
- 第五阶段：3 个审查代理产出 900 行反馈
- 第六阶段：3 个修订代理删减 38%（-3,934 行），新增 25+ 张 Mermaid 图表
- 第七阶段：1 个审计代理识别出 35 处完全复制，1 个清理代理将全部替换为伪代码
- 最终成品：6,271 行 / 约 400 页等量内容
